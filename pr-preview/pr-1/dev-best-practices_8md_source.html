    8 ## Coding "Dos" and "Don'ts": {#coding-dos-and-donts}
    9 
    10 ### Dos
    11  * Use static memory. See also [Static vs. Dynamic Memory](#static-vs-dynamic).
    12  * Select the priorities carefully.
    13  * Minimize stack usage with `DEVELHELP` and `CREATE_STACKTEST`.
    14  * Use threads to increase flexibility, modularity, and robustness by leveraging IPC.
    15  * Use unsigned or signed integer (`unsigned`, `int`, `size_t` or `ssize_t`) for loop variables wherever possible, but keep in mind that on some platforms an `int` has a width of only 16-bit. In general, you should avoid types like `uint8_t` for loop iterators as they will probably make it more expensive on some platforms.
    16  * Join and factor out parts of the code with existing code in RIOT, where it makes sense.
    17  * Check all `size/length` parameters when passing memory, e.g. using `sizeof(x)` or `strlen(x)` as appropriate. Make sure you don't use the wrong one with a pointer.
    18  * Make sure all code paths can be reached. Make sure there are no always `true/false` conditions.
    19  * Make sure all critical sections (`lock/unlock`, `acquire/release`, ...) are always closed on every code path.
    20  * Make sure return values are consistent with our API documentation.
    21  * Use `assert()` statements to check parameters rather than returning an error code at run-time, to keep the code size down.
    22  * Use the `DEBUG(...)` macro rather than `log_x(...)`
    23  * Declare all internal module variables and functions `static`
    24  * Make sure variables are reduced in scope as much as possible
    25  * Use an appropriate signedness in your variables
    26  * Make sure the variables are big enough to prevent overflow. Be aware that the code may run on platforms with different sizes of variables. For example, `int/unsigned` is only 16-bit on msp430 and avr8. If in doubt, use portable types.
    27  * Reduce the number of function calls as far as possible without duplicating code.
    28  * Use good judgement when using `static inline` functions and macros. If they are used in multiple places, is the increase in performance worth the penalty in code size?
    29  * Use memory judiciously in general. For example:
    30 ```c
    31 typedef enum {
    32  A,
    33  B,
    34  ...
    35 } foo_t;
    36 
    37 int bar(foo_t v)
    38 {
    39  int abc;
    40  ...
    41 
    42  switch(v) {
    43  case A:
    44  abc = 23;
    45  break;
    46  case B:
    47  abc = 42;
    48  break;
    49  ...
    50  }
    51  ...
    52 }
    53 
    54 /* VS */
    55 
    56 typedef enum {
    57  A = 23,
    58  B = 42,
    59  ...
    60 } foo_t;
    61 
    62 int bar(foo_t v) {
    63  int abc = v;
    64  ...
    65 }
    66 ```

    68 ### Don'ts
    69  * Don't use too many threads. Try not to use more than one thread per module. Don't create threads for one-time tasks.
    70  * Don't use the POSIX wrapper if implementing something from scratch.
    71  * Don't allocate big chunks of memory (for instance the IPC message queue) on the stack, but use rather static memory for that.
    72  * Don't over-provision memory.
    73  * Don't pass stack memory between different contexts unless you can prove conclusively that it won't be a problem.
    74  * Don't use enums for flags, because flags have a width in memory that is in most cases smaller than `sizeof(enum)` (most bitfields are 16 bits max, on most of our newer platforms, `sizeof(enum)` is however 32 bits). This results in every assignment needed to be cast to either `uint8_t` or `uint16_t`. With macros you don't need to cast since they are typeless. Making the enum packed makes its width unpredictable in terms of alignment issues, when used in struct.
    75  * Don't duplicate code from elsewhere in the RIOT code base, unless there is a very good reason to do so.
    76  * Don't duplicate code within your own code, unless there is a very good reason to do so. Use internal functions to this end.
    77  * Don't mix up logical and bitwise operations (`!` vs `~`, or `&&` vs `&`)

    79 ## Methodology: emulator first, target IoT hardware last! {#methodology}
    80 
    81 The below methodology is recommended, using well-known de facto standard tools from the FLOSS community that are compatible with RIOT. Using the below workflow improves time-to-running-code compared to typical IoT software workflows (which can be as retro as "LED-driven" debugging).
    82 
    83 0. For newbies, preliminaries are typically faster with the provisioned virtual environment setup, e.g. with **Vagrant**.
    84 1. To check your code, first use available **static analysis** as much as possible initially, which means (i) enable all compiler warnings and fix all problems found, then (ii) use a supported linter such as **cppcheck** to find bad coding patterns (i.e. code smells) and identify misuse of standard APIs.
    85 2. Next, use available **dynamic analysis** tools to find further defects while running the code on **RIOT native**, which means (i) running unit tests and integration tests on RIOT native emulator, and (ii) using **Valgrind** memcheck, as well as the **GCC stack smashing detection**, to detect and avoid undefined behavior due to invalid memory access.
    86 3. In case of networked applications or protocols, test **several instances of native** communicating via a virtual network mimicking the targeted scenario, which means (i) either using the default virtual full-mesh or other topologies configured via DESvirt, and (ii) using **Wireshark** to capture and analyze virtual network traffic, e.g. to ensure protocol packets are syntactically correct, and to observe network communication patterns.
    87 4. In case of incorrect behavior at this stage, analyze the system state for semantic errors on native using the standard debugger **gdb**, which allows virtually unlimited conditional breakpoints, record and replay, catchpoints, tracepoints and watchpoints.
    88 5. In case of suspected performance bottleneck, use performance profilers **gprof**, or else cachegrind, to identify precisely the bottlenecks.
    89 6. At this stage the implementation has proven bug-free on the native emulator. One can thus finally move on to hardware-in-the-loop, which means (i) flashing the binary on the targeted IoT hardware, typically using standard flasher **OpenOCD** or **edbg**, and (ii) using the **RIOT shell** running on the target IoT device(s) for easier debugging on the target hardware.
    90 7. In case the hardware is not available on-site, one can consider remotely flashing and testing the binary on supported open-access testbeds, e.g. [IoT-LAB](https://www.iot-lab.info) hardware is fully supported by RIOT.
    91 8. In case of failure, after analyzing the failure and attempting to fix the defect, go back to step 1 to make sure the fix did not itself introduce a new defect.

    93 ## Static vs. Dynamic Memory {#static-vs-dynamic}
    94 
    95 In your C program you have to decide where the memory you want to use comes from.
    96 There are two ways to get memory in your C code:
    97 
    98 1. Define static memory.
    99 2. Use dynamic memory (call `malloc()`/`free()` to get memory from the heap).
    100 
    101 Both ways have some drawbacks which are listed here.
    102 If you want to analyze the static memory consumption of your code you can use [otm](https://github.com/LudwigOrtmann/otm) or `make cosy`.
    103 
    104 ### Static memory
    105 * Access the memory in one operation O(1) â‡’ real time condition
    106 * Programmer needs to know the amount of memory on compile time
    107  * Leads to over and undersized buffers
    108 * Forces the programmer to think about the amount of need memory at compile time
    109 
    110 ### Dynamic memory
    111 * `malloc()` and `free()` are implemented in your `libc` (RIOT on ARM: `newlib`/`picolib`)
    112  * Runtime behavior is not predictable
    113 * Code can request the amount of memory it needs on runtime
    114 * On most platforms: the size of the heap is `sizeof(<RAM>)-sizeof(<static memory>)`
    115  * If you reduce your usage of static memory your heap gets bigger
    116 * On some platforms calling `free()` will not or not always make heap memory available again (see @ref oneway_malloc on MSP430)
    117 * Programmer needs to handle failed memory allocation calls at runtime
    118 * Static code analysis is unable to find errors regarding memory management
