    6 Introduction {#iomaps-intro}
    7 ============
    8 
    9 Arduino has popularized the concept of adding an abstraction layer over the
    10 MCU GPIOs and ADC multiplexer as "Digital Pins" and "Analog Pins". For historic
    11 reasons as well as to express appreciation for the introduction of concept, we
    12 prefix features and macros that map board pins to MCU pins with `ARDUINO`.
    13 
    14 @note This means, we also explicitly use the Arduino feature for boards that
    15  are neither official Arduino products nor compatible.
    16 
    17 I/O Mappings {#iomaps-mapping}
    18 ============
    19 
    20 All I/O mappings are provided by the `arduino_iomap.h` header file. This is
    21 provided either by the board directly (e.g. in
    22 `<RIOT git repo>/board/<board>/include/arduino_iomap.h`) or by the board
    23 family (e.g. in `board/common/<board family>/include/arduino_iomap.h`).
    24 
    25 To add support to new boards, start by copy-pasting the `arduino_iomap.h` from
    26 an existing board and adapt/extend the macros as needed. Finally, add the
    27 features to the `Makefile.features` and `KConfig` files of the board that
    28 indicate the presence of the added mappings.
    29 
    30 Digital Pins {#iomaps-mapping-gpio}
    31 ------------
    32 
    33 The feature `arduino_pins` is provided by boards that do have a mapping to
    34 digital pins. The GPIO for e.g. `D3` is provided as `ARDUINO_PIN_3`. The
    35 number of the highest digital pin is provided as `ARDUINO_PIN_LAST`. E.g. if
    36 `ARDUINO_PIN_LAST` is 42, digital pins `D0` to `D42` are typically provided.
    37 
    38 @warning Some boards do have "gaps" in the pin mapping. It could be that
    39  e.g. `ARDUINO_PIN_5` is defined, but `ARDUINO_PIN_4` is not.
    40 
    41 @note Often aliases are provided for analog pins in digital mode. E.g.
    42  to use the pin `A3` as digital pin, `ARDUINO_PIN_A3` can be used
    43  to avoid looking up what digital number that pin has.
    44 
    45 Analog Pins {#iomaps-mapping-adc}
    46 -----------
    47 
    48 The feature `arduino_analog` is provided by boards that do have a mapping of
    49 ADC lines to analog pins. E.g. the ADC line corresponding to pin `A5` would
    50 be `ARDUINO_A5`. The number of the highest analog pin is provided as
    51 `ARDUINO_ANALOG_PIN_LAST`. The macro `ARDUINO_ANALOG_PIN_LAST` is defined if
    52 and only if a mapping is provided (so it can be used to test for this feature).
    53 
    54 @warning Some boards do have "gaps" in the analog pin mapping. It could be
    55  that e.g. `ARDUINO_A3` is defined, but `ARDUINO_A2` is not.
    56 
    57 DAC Pins {#iomaps-mapping-dac}
    58 --------
    59 
    60 The feature `arduino_dac` is provided by boards that do have a mapping of
    61 DAC pins to DAC lines. E.g. the DAC line for the pin DAC1 would be
    62 `ARDUINO_DAC1`. The number of the highest DAC pin is provided as
    63 `ARDUINO_DAC_PIN_LAST`. The macro `ARDUINO_DAC_PIN_LAST` is defined if
    64 and only if a mapping is provided (so it can be used to test for this feature).
    65 
    66 @warning Some boards do have "gaps" in the analog pin mapping. It could be
    67  that e.g. `ARDUINO_DAC4` is defined, but `ARDUINO_DAC3` is not.
    68 
    69 PWM Pins {#iomaps-mapping-pwm}
    70 --------
    71 
    72 The feature `arduino_pwm` is provided by boards that do have a mapping of
    73 digital pins to PWM settings. E.g. the PWM device connected to the digital pin
    74 `D11` would be `ARDUINO_PIN_11_PWM_DEV` and the channel would be
    75 `ARDUINO_PIN_11_PWM_CHAN`. A PWM frequency for all PWM pins is defined as
    76 `ARDUINO_PWM_FREQU`.
    77 
    78 @warning Typically only few digital pins support PWM. For pins without PWM
    79  output no `ARDUINO_PIN_<NUM>_PWM_DEV` macro and no
    80  `ARDUINO_PIN_<NUM>_PWM_DEV` is defined.
    81 
    82 UART Device {#iomaps-mapping-uart}
    83 -----------
    84 
    85 The feature `arduino_uart` is provided by boards that do provide an UART device
    86 mapping. For the official Arduino boards and compatible boards, the
    87 `ARDUINO_UART_D0D1` macro refers to the UART device that uses the digital pins
    88 `D0` and `D1`.
    89 
    90 Please extend the table below to keep naming among boards of the same form
    91 factor consistent:
    92 
    93 | Form Factor | Macro Name | Description |
    94 |:------------- |:--------------------- |:--------------------------------- |
    95 | Arduino Nano | `ARDUINO_UART_D0D1` | UART on D0 (RXD) and D1 (TXD) |
    96 | Arduino Uno | `ARDUINO_UART_D0D1` | UART on D0 (RXD) and D1 (TXD) |
    97 | Arduino Mega | `ARDUINO_UART_D0D1` | UART on D0 (RXD) and D1 (TXD) |

    99 I²C Buses {#iomaps-mapping-i2c}
    100 ---------
    101 
    102 The feature `arduino_i2c` is provided by boards that do provide an I²C bus
    103 mapping.
    104 
    105 | Form Factor | Macro Name | Description |
    106 |:------------- |:--------------------- |:--------------------------------- |
    107 | Arduino Nano | `ARDUINO_I2C_NANO` | D18 (SDA) / D19 (SCL) |
    108 | Arduino Uno | `ARDUINO_I2C_UNO` | D18 (SDA) / D19 (SCL) |
    109 | Arduino Mega | `ARDUINO_I2C_UNO` | D20 (SDA) / D21 (SCL) |
    110 | Arduino Zero | `ARDUINO_I2C_UNO` | D20 (SDA) / D21 (SCL) |
    111 | Arduino Due | `ARDUINO_I2C_UNO` | D70 (SDA) / D71 (SCL) |
    112 
    113 The `ARDUINO_I2C_UNO` refers to the I²C bus next to the AREF pin (the topmost
    114 pins on header on the top right) of an Arduino Uno compatible board, e.g.
    115 such as the Arduino Mega2560. Even though the Arduino UNO, the Arduino MEGA2560,
    116 the Arduino Zero and the Arduino Zero all having the I²C bus at the exact same
    117 mechanical positions, the digital pin number of the I²C bus next to the AREF
    118 differs between the versions.
    119 
    120 SPI Buses {#iomaps-mapping-spi}
    121 ---------
    122 
    123 The feature `arduino_spi` is provided by boards that do provide an SPI bus
    124 mapping.
    125 
    126 | Form Factor | Macro Name | Description |
    127 |:------------- |:------------------------- |:------------------------------------- |
    128 | Arduino Nano | `ARDUINO_SPI_ISP` | The SPI on the ISP header |
    129 | Arduino Nano | `ARDUINO_SPI_D11D12D13` | D11 (MOSI) / D12 (MISO) / D13 (SCK) |
    130 | Arduino Uno | `ARDUINO_SPI_ISP` | The SPI on the ISP header |
    131 | Arduino Uno | `ARDUINO_SPI_D11D12D13` | D11 (MOSI) / D12 (MISO) / D13 (SCK) |
    132 | Arduino Mega | `ARDUINO_SPI_ISP` | The SPI on the ISP header |
    133 | Arduino Mega | `ARDUINO_SPI_D11D12D13` | D11 (MOSI) / D12 (MISO) / D13 (SCK) |
    134 
    135 @note The original AVR based Arduinos only have a single hard SPI bus which
    136  is only provided via the ISP header. Many modern Arduino compatible
    137  boards do not provide the ISP header and only have SPI on D11/D12/D13
    138  provided.

    140 Mechanical and Electrical Compatibility {#iomaps-shields}
    141 =======================================
    142 
    143 Modules implementing drivers for extension boards, a.k.a. shields, can express
    144 their mechanical and electrical requirements by depending on `arduino_shield_...`
    145 features. The following list of features currently exists:
    146 
    147 | Feature Name | Compatibility Claim |
    148 |
    149 | `arduino_shield_nano` | Board has side headers compatible with the Arduino Nano |
    150 | `arduino_shield_uno` | Board has side headers compatible with the Arduino UNO |
    151 | `arduino_shield_mega` | Board has side headers compatible with the Arduino MEGA |
    152 | `arduino_shield_isp` | Board has an ISP header |
    153 
    154 @note A board providing `arduino_shield_mega` **MUST** also provide
    155  `arduino_shield_uno`, as Arduino MEGA boards are backward compatible to
    156  shields for the Arduino UNO.
    157 
    158 E.g. a module that implements the W5100 based Ethernet shield would depend on
    159 both `arduino_shield_uno` and `arduino_shield_isp` for electrical compatibility
    160 as well as on `arduino_spi` for SPI bus mapping and `arduino_pins` for the CS
    161 pin mapping. This module could reuse the existing `w5100` driver and just
    162 supply the correct `w5100_params_t` using the I/O mappings.
