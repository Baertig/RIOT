    6 General Approach {#flashing-general}
    7 ================
    8 
    9 In general, flashing a board from RIOT is as straight forward as typing in a
    10 shell (with the application directory as current working directory):
    11 
    12 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
    13 make BOARD=<BOARD-TO-FLASH> flash
    14 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
    15 
    16 This will **rebuild** ***AND*** **flash** the application in the current working
    17 directory for board `<BOARD-TO-FLASH>`, using its default programming tool. If
    18 you want to use an alternative programming tool, say `stm32flash`, use:
    19 
    20 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
    21 make BOARD=<BOARD-TO-FLASH> PROGRAMMER=stm32flash flash
    22 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
    23 
    24 To flash without rebuilding use `flash-only` as target instead of `flash`.

    27 Supported Tools {#flashing-supported-tools}
    28 ===============
    29 
    30 RIOT supports plenty of flashing tools, that are below grouped into general
    31 flashing tools that support multiple MCU families, and specialized tools that
    32 only support one platform.
    33 
    34 Note that some programmers require additional configuration on a per board
    35 level or rely on features only available on some boards. Hence, a given board
    36 may not be supported by a programmer listed as supported for the platform of
    37 the board due to a missing board feature, bootloader, or similar.
    38 
    39 To ease use the programmers are given by the value to pass via
    40 `PROGRAMMER=<value>`, rather than the official spelling of the programmer.

    132 Programmer Configuration {#flashing-configuration}
    133 ========================
    134 
    135 This section will list additional configuration options to control the behavior
    136 of a programming tool, such as selecting the hardware adapter used for
    137 programming.
    138 
    139 OpenOCD Configuration {#flashing-configuration-openocd}
    140 ---------------------
    141 
    142 ### OPENOCD_DEBUG_ADAPTER
    143 
    144 `OPENOCD_DEBUG_ADAPTER` can be set via command line or as environment variable
    145 to use non-default flashing hardware.
    146 
    147 ### OPENOCD_RESET_USE_CONNECT_ASSERT_SRST
    148 
    149 `OPENOCD_RESET_USE_CONNECT_ASSERT_SRST` can be set via command line or as
    150 environment variable to `0` to disable resetting the board via the `SRST` line.
    151 This is useful when the `SRST` signal is not connected to the debug adapter or
    152 when using cheap ST-Link V2 clones with broken `SRST` output. Note that it may
    153 not be possible to attach the debugger while the MCU is in deep sleeping mode.
    154 If this is set to `0` by the user, the user may need a carefully timed reset
    155 button press to be able to flash the board.
    156 
    157 ### OPENOCD_PRE_FLASH_CMDS
    158 
    159 `OPENOCD_PRE_FLASH_CMDS` can be set as environment variable to pass additional
    160 commands to OpenOCD prior to flashing, e.g. to disable flash write protection.
    161 
    162 ### OPENOCD_PRE_VERIFY_CMDS
    163 
    164 `OPENOCD_PRE_VERIFY_CMDS` can be set as environment variable to pass additional
    165 flags to OpenOCD prior to verifying the flashed firmware. E.g. this is used
    166 in the `pba-d-01-kw2x` to disable the watchdog to prevent it from disrupting
    167 the verification process.
    168 
    169 ### OPENOCD_PRE_FLASH_CHECK_SCRIPT
    170 
    171 `OPENOCD_PRE_FLASH_CHECK_SCRIPT` can be set via command line or as
    172 environment variable to execute a script before OpenOCD starts flashing. It is
    173 used for Kinetis boards to prevent bricking a board by locking the flash via
    174 magic value in the flash configuration field protection bits.
    175 
    176 The script is expected to exit with code `0` if flashing should resume, or with
    177 exit code `1` if flashing should be aborted.
    178 
    179 ### OPENOCD_CONFIG
    180 
    181 `OPENOCD_DEBUG_ADAPTER` can be set via command line or as environment variable
    182 to use non-default OpenOCD configuration file.
    183 
    184 ### OPENOCD_TRANSPORT
    185 
    186 `OPENOCD_TRANSPORT` can be set via command line or as environment variable to
    187 select a non-default transport protocol. E.g. to use JTAG rather than SWD for a
    188 board that defaults to SWD use:
    189 
    190 ```
    191 make PROGRAMMER=openocd OPENOCD_TRANSPORT=jtag
    192 ```
    193 
    194 Note that OpenOCD configuration file of a given board may only support SWD or
    195 JTAG. Also JTAG requires more signal lines to be connected compared to SWD and
    196 some internal programmers only have the SWD signal lines connected, so that
    197 JTAG will not be possible.
    198 
    199 stm32flash Configuration {#flashing-configuration-stm32flash}
    200 ------------------------
    201 
    202 It is possible to automatically boot the STM32 board into the in-ROM bootloader
    203 that `stm32flash` communicates with for flashing by connecting the RST pin to
    204 DTR and the BOOT pin (or BOOT0 for STM32 MCU families with BOOT0 and BOOT1 pins)
    205 to RTS of the TTL adapter. In addition, set `STM32FLASH_RESET` to `1` via
    206 environment or command line to actually issue a reset with BOOT (or BOOT0)
    207 pulled high prior flashing to enter the bootloader, and a second reset with BOOT
    208 (or BOOT0) pulled low to reboot into the application. `STM32FLASH_RESET`
    209 defaults to `0` as of know, as with `PROGRAMMER=stm32flash STM32FLASH_RESET=1`
    210 additional terminal flags are set, so that `make term` doesn't accidentally
    211 keeps the reset signal pulled low or boot the board into the bootloader.
    212 
    213 The TTL adapter this was tested with had inverted RTS and DTR signal. By setting
    214 `STM32FLASH_RESET_INVERT` to `1` RIOT will assume RTS and DTR signals to be
    215 inverted, by setting it to `0` non-inverted signals will be generated. As of
    216 now, `STM32FLASH_RESET_INVERT` is by default `1`. This may change if it
    217 becomes evident that non-inverted TTL adapters are in fact more common than
    218 inverted adapters.

    220 MSPDEBUG Configuration {#flashing-configuration-mspdebug}
    221 ----------------------
    222 
    223 All options can be passed as environment variables or as make arguments.
    224 All options except for `DEBUGSERVER_PORT` apply to both debugging and flashing
    225 alike.
    226 
    227 `MSPDEBUG_PROGRAMMER` is used to set the hardware programmer/debugger to use
    228 for programming and debugging. See `mspdebug --help` or `man mspdebug` for a
    229 list of programmers.
    230 
    231 `MSPDEBUG_PROTOCOL` is used to specify the debugger protocol. It is typically
    232 set by the board used. Only JTAG and Spi-Bi-Wire are supported.
    233 
    234 `MSPDEBUG_TTY` is used to connect via TTY interface instead of directly via
    235 USB to the debugger. Usually, this is not required.
    236 
    237 `DEBUG_ADAPTER_ID` is used to select the debugger/programmer by its serial. If
    238 not set, `mspdebug` will select the first device with matching vendor and
    239 product ID. Unless multiple debuggers of the same type are connected, this
    240 options is typically not needed.
    241 
    242 `DEBUGSERVER_PORT` is used to specify the TCP port to listen for GDB to
    243 connect to. It defaults to 2000.

    245 Handling Multiple Boards With UDEV-Rules {#multiple-boards-udev}
    246 ========================================
    247 
    248 When developing and working with multiple boards the default `PORT`
    249 configuration for a particular board might not apply anymore so `PORT` will need
    250 to be specified whenever calling `make term/test`. This can also happen if
    251 multiple `DEBUGGERS/PROGRAMMERS` are present so `DEBUG_ADAPTER_ID` will also
    252 need to be passed. Keeping track of this will become annoying.
    253 
    254 One way of handling this is to use `udev` rules to define `SYMLINKS` between the
    255 boards serial port (`riot/tty-<board-name>`) and the actual serial port
    256 (dev/ttyACM* or other). With this we can query the rest of the boards serial
    257 `dev` information (`DEBUG_ADAPTER_ID`, `PORT`, etc.) to always flash and open a
    258 terminal on the correct port.
    259 
    260 Procedure:
    261 
    262 - use `udevadm info /dev/ttyACM0` to query the udev database for information on
    263  device on port `/dev/ttyACM0`.
    264 
    265  or use `udevadm info --attribute-walk --name /dev/ttyACM0` for more detailed
    266  output when the first level of information isn't enough
    267 
    268 - create a udev rule with information of the device and one parent to create a
    269  matching rule in `/etc/udev/rules.d/70-riotboards.rules`.
    270 
    271 ~~~~~~~~~~~~~~~~~~~
    272  # samr21-xpro
    273  SUBSYSTEM=="tty", SUBSYSTEMS=="usb", ATTRS{idVendor}=="03eb", \
    274  ATTRS{idProduct}=="2111", ATTRS{manufacturer}=="Atmel Corp.", \
    275  ATTRS{serial}=="ATML2127031800004957", SYMLINK+="riot/tty-samr21-xpro"
    276 ~~~~~~~~~~~~~~~~~~~
    277 
    278 - reload rules: `udevadm control --reload-rules`
    279 
    280 - Boards `PORT` are symlinked to /dev/riot/tty-`board-name`.
    281 
    282 - Create a `makefile.pre` that will query the real `PORT` and the
    283  `DEBUG_ADAPTER_ID` from the `SYMLINK` info
    284 
    285 ~~~~~~~~~~~~~~~~~~~
    286  PORT = /dev/riot/tty-$(BOARD)
    287  DEBUG_ADAPTER_ID = $(\
    288  shell udevadm info -q property $(PORT) |\
    289  sed -n ’/ID_SERIAL_SHORT/ {s/ID_SERIAL_SHORT=//p}’)
    290 ~~~~~~~~~~~~~~~~~~~
    291 
    292 - You can now add `makefile.pre` to `RIOT_MAKEFILES_GLOBAL_PRE` as an environment
    293  variable or on each `make` call:
    294 
    295 ~~~~~~~~~~~~~~~~~~~
    296  $ RIOT_MAKEFILES_GLOBAL_PRE=/path/to/makefile.pre make -C examples/basic/hello-world flash term
    297 ~~~~~~~~~~~~~~~~~~~
    298 
    299 @note if set as an environment variable it would be a good idea to add a
    300  variable to enable/disable it, e.g:
    301 
    302 ~~~~~~~~~~~~~~~~~~~
    303 ifeq (1,$(ENABLE_LOCAL_BOARDS))
    304  PORT = /dev/riot/tty-$(BOARD)
    305  DEBUG_ADAPTER_ID = $(\
    306  shell udevadm info -q property $(PORT) |\
    307  sed -n ’/ID_SERIAL_SHORT/ {s/ID_SERIAL_SHORT=//p}’)
    308 endif
    309 ~~~~~~~~~~~~~~~~~~~
    310 
    311 Handling Multiple Versions of the Same BOARD
    312 --------------------------------------------
    313 
    314 The above procedure works fine when handling different boards, but not
    315 multiple times the same board, e.g: multiple `samr21-xpro`.
    316 
    317 An option for this would be to add an identifier of that board to the mapped
    318 `riot/tty-*`, there are multiple ways of handling this but in the end it means
    319 having a way to identify every copy.
    320 
    321 Another way would be to map the `DEBUG_ADAPTER_ID` in the name:
    322 
    323 ~~~~~~~~~~~~~~~~~~~
    324 SYMLINK+="riot/node-$attr{serial}
    325 ~~~~~~~~~~~~~~~~~~~
    326 
    327 But it will require to know in advance the serial number of each board you want
    328 to use. Another option would be to add some kind of numbering and defining
    329 multiple symlinks for each board. e.g. for `samr21-xpro` number `n`:
    330 
    331 ~~~~~~~~~~~~~~~~~~~
    332  # samr21-xpro
    333  SUBSYSTEM=="tty", SUBSYSTEMS=="usb", ATTRS{idVendor}=="03eb", \
    334  ATTRS{idProduct}=="2111", ATTRS{manufacturer}=="Atmel Corp.", \
    335  ATTRS{serial}=="ATML2127031800004957", SYMLINK+="riot/tty-samr21-xpro", \
    336  SYMLINK+="riot/tty-samr21-xpro-n"
    337 ~~~~~~~~~~~~~~~~~~~
    338 
    339 Then, when flashing, the number can be specified and the parsing adapted:
    340 
    341 ~~~~~~~~~~~~~~~~~~~
    342  ifneq(,$(BOARD_NUM))
    343  PORT = /dev/riot/tty-$(BOARD)-$(BOARD_NUM)
    344  else
    345  PORT = /dev/riot/tty-$(BOARD)
    346  endif
    347  DEBUG_ADAPTER_ID = $(\
    348  shell udevadm info -q property $(PORT) |\
    349  sed -n ’/ID_SERIAL_SHORT/ {s/ID_SERIAL_SHORT=//p}’)
    350 ~~~~~~~~~~~~~~~~~~~
    351 
    352 ~~~~~~~~~~~~~~~~~~~
    353  BOARD=samr21-xpro BOARD_NUM=n make flash term
    354 ~~~~~~~~~~~~~~~~~~~
    355 
    356 In the end, this would be the same as using the serial, but a simple number might
    357 be easier to handle.
    358 
    359 Notes
    360 -----
    361 Udev only parses SUBSYSTEM and one parent. For others, we will rely on ENV
    362 variables defined by 60-serial.rules
    363 
    364 So the current filename should be higher than 60-serial.rules
    365 
    366 If for some reason re-writing the serial is needed there is a windows tool:
    367  https://remoteqth.com/wiki/index.php?page=How+to+set+usb+device+SerialNumber

    377 Handling Multiple Boards Without UDEV-Rules {#multiple-boards-no-udev}
    378 ===========================================
    379 
    380 This is a simpler approach to the above mentioned issue. The solution here only
    381 uses a makefile script for selecting the debugger and serial port. No
    382 administrative privileges (e.g. to configure Udev) are required.
    383 
    384 One of the limitations of the solution described here is that it currently
    385 doesn't work with multiple boards of the same type. This limitation is a
    386 limitation of the script and not of the mechanism used, it is possible to adapt
    387 the script to support multiple boards of the same type. This modification is
    388 left as an exercise to the reader.
    389 
    390 The following Make snippet is used:
    391 
    392 ~~~~~~~~~~~~~~~~~~~
    393  LOCAL_BOARD_MAP ?= 1
    394 
    395  # Adapt this list to your board collection
    396  SERIAL_nucleo-f103rb ?= 066BFF343633464257254156
    397  SERIAL_same54-xpro ?= ATML2748051800005053
    398  SERIAL_samr21-xpro ?= ATML2127031800008360
    399  SERIAL_nrf52dk ?= 000682223007
    400 
    401  ifeq (1,$(LOCAL_BOARD_MAP))
    402 
    403  # Retrieve the serial of the selected board
    404  BOARD_SERIAL = $(SERIAL_$(BOARD))
    405 
    406  # Check if there is a serial for the board
    407  ifneq (,$(BOARD_SERIAL))
    408 
    409  # Set the variables used by various debug tools to the selected serial
    410  SERIAL ?= $(BOARD_SERIAL)
    411  DEBUG_ADAPTER_ID ?= $(BOARD_SERIAL)
    412  JLINK_SERIAL ?= $(BOARD_SERIAL)
    413 
    414  # Use the existing script to grab the matching /dev/ttyACM* device
    415  PORT ?= $(shell $(RIOTTOOLS)/usb-serial/ttys.py --most-recent --format path --serial $(SERIAL))
    416  endif
    417  endif
    418 ~~~~~~~~~~~~~~~~~~~
    419 
    420 The array of board serial numbers has to be edited to match your local boards.
    421 The serial numbers used here is the USB device serial number as reported by
    422 the debugger hardware. With the `make list-ttys` it is reported as the 'serial':
    423 
    424 ~~~~~~~~~~~~~~~~~~~
    425 $ make list-ttys
    426 path | driver | vendor | model | model_db | serial | ctime
    427 -------------|---------|--------------------------|--------------------------------------|-----------------------|--------------------------|---------
    428 /dev/ttyUSB0 | cp210x | Silicon Labs | CP2102 USB to UART Bridge Controller | CP210x UART Bridge | 0001 | 15:58:13
    429 /dev/ttyACM1 | cdc_acm | STMicroelectronics | STM32 STLink | ST-LINK/V2.1 | 0671FF535155878281151932 | 15:58:04
    430 /dev/ttyACM3 | cdc_acm | Arduino (www.arduino.cc) | EOS High Power | Mega ADK R3 (CDC ACM) | 75230313733351110120 | 15:59:57
    431 /dev/ttyACM2 | cdc_acm | SEGGER | J-Link | J-Link | 000683475134 | 12:41:36
    432 ~~~~~~~~~~~~~~~~~~~
    433 
    434 When the above make snippet is included as `RIOT_MAKEFILES_GLOBAL_PRE`, the
    435 serial number of the USB device is automatically set if the used board is
    436 included in the script. This will then ensure that the board debugger is used
    437 for flashing and the board serial device is used when starting the serial
    438 console.
    439 
    440 It supports command line parameters to filter by vendor name, model name, serial
    441 number, or driver. In addition, the `--most-recent` argument will only print the
    442 most recently added interface (out of those matching the filtering by vendor,
    443 model, etc.). The `--format path` argument will result in only the device path
    444 being printed for convenient use in scripts.
    445 
    446 Handling Multiple Boards: Simplest Approach {#multiple-boards-simple}
    447 ===========================================
    448 
    449 Passing `MOST_RECENT_PORT=1` as environment variable or as parameter to
    450 make will result in the most recently connected board being preferred over the
    451 default PORT for the selected board.
    452 
    453 For some boards `TTY_BOARD_FILTER` is provided, which filters TTYs e.g. by
    454 vendor or model to only considered TTYs that actually may belong to the selected
    455 board. E.g. for Nucleo boards this is `--model 'STM32 STLink'`, as they all use
    456 an integrated STLink as programmer. As long as only one TTY is provided from an
    457 STLink, this will reliably select the correct TTY for an Nucleo regardless of
    458 which TTY was most recently connected. Some boards even provide info that
    459 allows to always reliably identify them correctly (e.g. the firmware on the
    460 ATmega16U2 used as USB to UART converted on Arduino Mega2560 will provide
    461 identification data unique to that board).
    462 
    463 Adding Board Filters
    464 --------------------
    465 
    466 After connecting as many variants of the board you target (and maybe some others
    467 to test that the filter actually filters out non-matching boards). Then first
    468 run `./dist/tools/usb-serial/ttys.py` without arguments and study the output.
    469 When a genuine Arduino Mega 2560, a genuine Arduino Mega ADK (a variant of the
    470 Mega 2560),a cheap Arduino Mega 2560 clone, a BBC micro:bit v2, and a
    471 Nucleo F767-ZI are connected, the following output is shown:
    472 
    473 path | driver | vendor | model | model_db | serial | ctime | iface_num
    474 -------------|---------|--------------------------|--------------------------------------|------------------------------------------------------|--------------------------------------------------|----------|----------
    475 /dev/ttyACM0 | cdc_acm | Arduino (www.arduino.cc) | 0042 | Mega 2560 R3 (CDC ACM) | 857353134333519002C1 | 12:13:55 | 0
    476 /dev/ttyACM1 | cdc_acm | Arduino (www.arduino.cc) | EOS High Power | Mega ADK R3 (CDC ACM) | 75230313733351110120 | 15:59:57 | 0
    477 /dev/ttyACM2 | cdc_acm | STMicroelectronics | STM32 STLink | ST-LINK/V2.1 | 0670FF535155878281123912 | 10:00:39 | 2
    478 /dev/ttyACM3 | cdc_acm | Arm | BBC micro:bit CMSIS-DAP | ARM mbed | 99053602000528334c41b84da1f2f09d000000006e052820 | 12:21:03 | 1
    479 /dev/ttyUSB0 | cp210x | Silicon Labs | CP2102 USB to UART Bridge Controller | CP2102/CP2109 UART Bridge Controller [CP210x family] | 0001 | 16:57:27 | 0
    480 
    481 Now we add arguments to the invocation of `ttys.py` to filter the list e.g.
    482 by model, vendor etc. (note: as regex!) ideally until only the target boards
    483 are listed. Some boards do not provide enough information to e.g. tell them
    484 apart from other boards using the same USB to UART bridge or the same debugger.
    485 In that case we have to live with some "bycatch".
    486 
    487 In the case of the Arduino Mega 2560 the parameters
    488 `--vendor 'Arduino' --model-db 'Mega 2560|Mega ADK'` will narrow down the
    489 list to only show the genuine Arduino Mega versions. Se we add to the
    490 `Makefile.include` in `boards/arduino-mega2560`:
    491 
    492 ```Makefile
    493 TTY_BOARD_FILTER := --vendor 'Arduino' --model-db 'Mega 2560|Mega ADK'
    494 ```
    495 
    496 Note that also matching the `R3` in `Mega 2560 R3` would prevent matching older
    497 or newer revisions than R3, so we don't add that to the regex.
    498 
    499 Advances Board Filters
    500 ----------------------
    501 
    502 In most cases, just adding a simple `TTY_BOARD_FILTER` is sufficient. If we
    503 however have wildly different flavors of the same board (e.g. genuine Arduino
    504 Mega 2560 with an ATmega16U2 and clones with a cheap USB to UART bridge) that we
    505 all want to support, we have to instead provide a `TTY_SELECT_CMD` that prints
    506 the path to and the serial of the TTY (separated by a space) and exists with
    507 `0` if a TTY was found, or that exists with `1` and prints nothing when no TTY
    508 was found. We can still use the `ttys.py` script to detect all Arduino Mega
    509 2560 versions: We first try to detect a genuine Arduino Mega and fall back to
    510 selecting cheap USB UART bridges when that fails using the `||` shell operator:
    511 
    512 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
    513  TTY_SELECT_CMD := $(RIOTTOOLS)/usb-serial/ttys.py \
