    1 Creating an application {#creating-an-application}
    2 =======================
    3 
    4 [TOC]
    5 
    6 To create your own application you need to create a directory containing one or
    7 multiple C file(s) with your source code and a Makefile. A template Makefile is
    8 available in the `dist` folder of the
    9 [RIOT repository](https://github.com/RIOT-OS/RIOT).
    10 
    11 The main function {#the-main-function}
    12 =================
    13 After the board is initialized, RIOT starts two threads: the idle thread and
    14 the main thread. The idle thread has the lowest priority and will run whenever
    15 no other thread is ready to run. It will automatically use the lowest possible
    16 power mode for the device. The main thread - configured with a default priority
    17 that is right in the middle between the lowest and the highest available
    18 priorities - is the first thread that runs and calls the `main()` function.
    19 This function needs to be defined in the source code of the application
    20 (typically located in `main.c`).
    21 
    22 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.c}
    23 #include <stdio.h>
    24 
    25 int main(void)
    26 {
    27  puts("Hello World!");
    28  return 0;
    29 }
    30 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

    32 The application's Makefile {#the-applications-makefile}
    33 ==========================
    34 The minimal Makefile {#the-minimal-makefile}
    35 --------------------
    36 At minimum the Makefile of an application (see @ref getting-started) needs to
    37 define the following macros:
    38  * `APPLICATION`: should contain the name of your application
    39  * `RIOTBASE`: specifies the path to your copy of the RIOT repository (note
    40  that you may want to use `$(CURDIR)` here, to give a relative path)
    41 
    42 The `BOARD` macro is also required and recommended to be set to `native` by
    43 default, but is recommended to be overridable with the `?=` operator.
    44 Additionally, it is required to include the `Makefile.include` from the
    45 `RIOTBASE`.
    46 
    47 ~~~~~~~~~~~~~~~~~~~~~~~~~~~ {.mk}
    48 # Set the name of your application:
    49 APPLICATION = foobar
    50 
    51 # If no BOARD is found in the environment, use this default:
    52 BOARD ?= native
    53 
    54 # This has to be the absolute path to the RIOT base directory:
    55 RIOTBASE ?= $(CURDIR)/../../RIOT
    56 
    57 include $(RIOTBASE)/Makefile.include
    58 ~~~~~~~~~~~~~~~~~~~~~~~~~~~

    60 How to handle unsupported boards? {#handle-unsupported-boards}
    61 ---------------------------------
    62 Sometimes it is necessary to exclude boards because they don't provide a
    63 required functionality or don't have sufficient memory. RIOT's build system
    64 looks for the macros `BOARD_BLACKLIST`, `BOARD_WHITELIST`, and
    65 `BOARD_INSUFFICIENT_MEMORY`. Any board name that is not included in
    66 `BOARD_WHITELIST` will issue a message that one has to expect errors if they
    67 build the application for the board referred by that name. The list can also be
    68 used by a CI system to not build the application for this board at all. A board
    69 that is included in `BOARD_BLACKLIST` will show the same behavior. The build
    70 system evaluates `BOARD_WHITELIST` first and then `BOARD_BLACKLIST`. The
    71 `BOARD_INSUFFICIENT_MEMORY` macro is similar to `BOARD_BLACKLIST` but will
    72 build in any case. A CI system can use the information provided by the
    73 `BOARD_INSUFFICIENT_MEMORY` macro to skip the linking step in the build
    74 process, since some linkers will issue an error if the code won't fit the
    75 target board's flash memory or RAM.

    77 Including modules {#including-modules}
    78 -----------------
    79 By default a RIOT application comprises only of the applications' code itself,
    80 the kernel, and platform specific code. In order to use additional modules,
    81 such as a particular [device driver](@ref drivers) or [a system library](@ref sys)
    82 (including [networking capabilities](@ref net)), you have to append the modules'
    83 names to the USEMODULE variable. For example, to build an application using the
    84 SHT11 temperature sensor and UDP/IPv6 functionalities of the GNRC network stack,
    85 your Makefile needs to contain these lines:
    86 
    87 ~~~~~~~~~~~~~~~~~~~~~~~~~~~ {.mk}
    88 USEMODULE += sht11
    89 USEMODULE += gnrc_ipv6_default
    90 USEMODULE += gnrc_udp
    91 ~~~~~~~~~~~~~~~~~~~~~~~~~~~
    92 
    93 Modules typically pull in all required dependencies.
    94 
    95 ## Including source files in subfolders
    96 
    97 By default, all source files in an application's (or any RIOT module's) directory
    98 are automatically compiled as part of the application. In order to organize source
    99 code in a directory structure, two different approaches can be used:
    100 
    101 1. Make each subdirectory a separate RIOT module with a unique name inside its
    102 Makefile, either by adding the directory's path to `DIRS` or with the [out-of-tree
    103 module support](#external-modules).
    104 2. Add the source files within subdirectories to `SRC`, either explicitly or with
    105 Makefile wildcards.
    106 
    107 Both approaches are illustrated and explained in `examples/basic/subfolders`.

    110 # Helper tools
    111 
    112 To help you start writing an application within RIOT, the build system provides
    113 the `generate-example` `generate-test` make targets. These targets are wrappers
    114 around the [riotgen](https://pypi.org/project/riotgen/) command line tool and
    115 are helpful when starting to implement an application: all required files are
    116 generated with copyright headers, doxygen groups, etc, so you can concentrate
    117 on the module implementation.
    118 For applications, the `Makefile` is generated with the dependencies (modules,
    119 packages, required features) included.
    120 
    121 **Usage:**
    122 
    123 To generate an example application, e.g in the `examples` directory, from the
    124 RIOT base directory, run:
    125 ```
    126 make generate-example
    127 ```
    128 
    129 To generate a test application, e.g in the `tests` directory, from the
    130 RIOT base directory, run:
    131 ```
    132 make generate-test
    133 ```
    134 
    135 Then answer a few questions about the application:
    136 - Application name: enter a name for your application. It will be used as both
    137  the name of the application directory under `examples` or `tests` and by
    138  the build system module (set in the `APPLICATION` variable).
    139 - Application brief description: Describe in one line what is this application
    140  about.
    141 - Target board: select the default target board. The value is set to `native`
    142  by default.
    143 - Modules: give the list of dependency modules, separated by commas. For
    144  example: ztimer,fmt
    145 - Packages: give the list of dependency packages, separated by commas.
    146 - Features required: give the list of CPU features (`periph_*`, etc) required
    147  by the application, all separated by commas.
    148 
    149 Other global information (author name, email, organization) should be retrieved
    150 automatically from your git configuration.
    151 
    152 Once completed, the application files are either located in
    153 `examples/<application name>` or `tests/<application name>` depending on the
    154 target used.
    155 
    156 **Testrunner:** when using the `make generate-test`, you can also automatically
    157 add a testrunner Python script. Just answer 'y' when prompted.

    159 # Creating an out of tree application structure
    160 
    161 Applications written for RIOT do not have to reside in the RIOT tree. Out of
    162 tree applications, modules and boards are supported.
    163 
    164 For a full application with custom board and modules, the following directory
    165 tree can be used:
    166 
    167 ```
    168 ├── apps
    169 │   └── my_app
    170 │   └── Makefile
    171 ├── boards
    172 │   └── my_board
    173 ├── modules
    174 │   └── my_module
    175 │   ├── include
    176 │   │   └── my_module.h
    177 │   ├── Makefile
    178 │   ├── Makefile.include
    179 │   └── my_module.c
    180 └── RIOT
    181 ```
    182 
    183 In this example tree, the `apps` directory contains a collection of applications
    184 for the project. The modules directory could contain extra modules for the
    185 applications.
    186 
    187 The make file inside the application needs at least the following as bare minimum:
    188 
    189 ```
    190 APPLICATION = my_app
    191 PROJECT_BASE ?= $(CURDIR)/../..
    192 RIOTBASE ?= $(PROJECT_BASE)/RIOT
    193 
    194 # Optionally, provide paths to where external boards and/or modules
    195 # reside, so that they can be included in the app
    196 EXTERNAL_MODULE_DIRS += $(PROJECT_BASE)/modules
    197 EXTERNAL_BOARD_DIRS += $(PROJECT_BASE)/boards
    198 
    199 include $(RIOTBASE)/Makefile.include
    200 ```
    201 
    202 The `RIOTBASE` variable tells the build system where to find the RIOT source
    203 tree and to need to point to the RIOT source tree used for the application for
    204 the application to work.
    205 
    206 The RIOT directory contains the sources of RIOT here. This can be either a
    207 direct checkout of the sources or a git submodule, whichever has your
    208 preference.
    209 
    210 If your project has separate modules or separate boards, these can be contained
    211 inside a modules os boards directory. The RIOT build system has both
    212 `EXTERNAL_MODULE_DIRS` and `EXTERNAL_BOARD_DIRS` variables to specify
    213 directories that contain extra modules and extra boards.
    214 
    215 ## External Boards
    216 
    217 External boards can be ported in an identical way as porting a regular board to
    218 RIOT, see @ref porting-boards.
    219 
    220 One approach can be to copy over an existing board and modify it to suit the
    221 needs. Boards in the RIOT tree can be included and used as dependency in the
    222 custom boards. In case you connect additional hardware to an upstream board
    223 (such as e.g. an Arduino shield) or you require a different hardware
    224 configuration (e.g. configuring some of the pins configured as ADC as
    225 additional PWM outputs instead) a copy of the upstream board that is then
    226 customized to the application needs is the best course of action.
    227 
    228 ## External Modules {#external-modules}
    229 
    230 Similar to the external boards, external modules can be written in a similar way
    231 as regular in-tree modules.
    232 
    233 One modification is the include directory inside the module directory. For this
    234 include directory to be added to the include path during compilation, the
    235 following snippet is required in the modules `Makefile.include`:
    236 
    237 ```
    238 USEMODULE_INCLUDES_my_module := $(LAST_MAKEFILEDIR)/include
    239 USEMODULE_INCLUDES += $(USEMODULE_INCLUDES_my_module)
    240 ```
    241 
    242 Note that the make variable (here `USEMODULE_INCLUDES_my_module`) must be unique
    243 for every module to make this work. Including the module name here is usually
    244 sufficient.
    245 
    246 ## Extra Makefile Scaffolding
    247 
    248 A bit of extra, but optional, Makefile scaffolding can help to keep the project
    249 easy to maintain. An extra `Makefile.include` in the root directory of the
    250 project that sets the necessary variables can help to deduplicate settings.
    251 This includes the `RIOTBASE` variable and the include to the RIOT
    252 `Makefile.include`:
    253 
    254 ```
    255 EXTERNAL_MODULE_DIRS += $(PROJECT_BASE)/modules
    256 EXTERNAL_BOARD_DIRS += $(PROJECT_BASE)/boards
    257 RIOTBASE ?= $(PROJECT_BASE)/RIOT
    258 include $(RIOTBASE)/Makefile.include
    259 ```
    260 Applications then just have to set the `PROJECT_BASE` variable and include this
    261 makefile and don't have to each add the external board and module directories.
    262 
    263 The application makefile would then look like this:
    264 ```
    265 APPLICATION = my_app
    266 PROJECT_BASE ?= $(CURDIR)/../..
    267 include $(PROJECT_BASE)/Makefile.include
    268 ```
