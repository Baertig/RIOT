    6 Downloading RIOT code {#downloading-riot-code}
    7 =====================
    8 You can obtain the latest RIOT code from
    9 our [Github](https://github.com/RIOT-OS/) repository either by
    10 [downloading the latest tarball](https://github.com/RIOT-OS/RIOT/releases) or
    11 by cloning the [git repository](https://github.com/RIOT-OS/RIOT).
    12 
    13 In order to clone the RIOT repository, you need the
    14 [Git revision control system](http://git-scm.com/) and run the following
    15 command:
    16 
    17 ~~~~~~~~ {.sh}
    18 git clone https://github.com/RIOT-OS/RIOT.git
    19 ~~~~~~~~

    24 Required Software for Development {#setting-up-a-toolchain}
    25 -----------------------------------------------------------
    26 
    27 A set of common tools and a toolchain for the hardware you target needs to be installed first.
    28 
    29 ### Choosing an Operating System for the Development PC
    30 
    31 Most of the RIOT OS developers are using Linux on their development PCs, so you can expect the
    32 most streamlined experience here. Other POSIX-compliant OSes such as the various BSD flavours
    33 will also be fine - however, we rely on users to report bugs regarding tooling incompatibilities
    34 here. So expect occasional issues for the development branch and please help testing during the
    35 feature freeze period, if you develop on macOS or BSD.
    36 
    37 Windows users can refer to [this guide][dev-setup-windows] to
    38 [setup the development environment][dev-setup-windows] on Windows.
    39 
    40 [dev-setup-windows]: https://github.com/RIOT-OS/RIOT/tree/master/doc/guides/setup-windows
    41 
    42 Native development on macOS machines is not officially supported. What works well is using Linux
    43 in a virtual machine, but at much lower performance than running Linux natively. We also offer Docker images.
    44 For development on Windows, using the
    45 [Windows Subsystem for Linux (WSL)](https://en.wikipedia.org/wiki/Windows_Subsystem_for_Linux)
    46 is a good option
    47 ([installation instructions here](https://docs.microsoft.com/en-us/windows/wsl/install)), but it
    48 is (as of October 2021) not possible to directly access USB devices from Linux. As a result,
    49 accessing the serial connection to a board running RIOT, flashing it, and on-chip-debugging from WSL
    50 will not be possible. (It is possible to pass through the file system of USB storage device. This
    51 should allow flashing boards that have an [UF2 compatible](https://github.com/Microsoft/uf2) from
    52 within WSL, but this has not been tested yet.) Hence, WSL users will have to use native Windows
    53 tools for accessing the serial connection and flashing the board.

    55 ### Common Tools
    56 
    57 The following tools are required or useful regardless of the architecture and board you are
    58 developing for:
    59 
    60 * Essential system development tools (GNU Make GCC, standard C library headers)
    61 * git
    62 * GDB in the multiarch variant (alternatively: install for each architecture you target the
    63  corresponding GDB package)
    64 * unzip or p7zip
    65 * wget or curl
    66 * python3
    67 * pyserial (linux distro package often named python3-serial or py3-serial)
    68 * Doxygen for building the documentation
    69 
    70 @note For each architecture a default tool for flashing and on-chip debugging is listed below - in
    71  most cases OpenOCD. However, some boards use different tools, e.g. because a bootloader is
    72  installed that allows flashing via the serial interface. Check the board documentation for any
    73  details on this. If that documentation contains no info about a flashing tool, the default
    74  tool for its architecture is used.
    75 
    76 For example, in Ubuntu the above tools can be installed with the following command:
    77 
    78  sudo apt install git gcc-arm-none-eabi make gcc-multilib libstdc++-arm-none-eabi-newlib openocd gdb-multiarch doxygen wget unzip python3-serial
    79 
    80 @details Running `BOARD=<INSERT_TARGET_BOARD_HERE> make info-programmers-supported` in your
    81  application folder lists the programmers supported by RIOT for the given board.
    82 
    83 ### Architecture: ARM7 and ARM Cortex M*
    84 
    85 * GCC, binutils, and newlib for `arm-none-eabi`
    86  * Alternatively: Install docker and export `BUILD_IN_DOCKER=1`
    87 * OpenOCD for debugging/flashing (most boards)
    88  * Some boards use UF2 based bootloaders, which require auto-mounting to work with `make flash`
    89  * Some boards default to using J-Link for flashing/debugging. Either install that or export
    90  `PROGRAMMER=openocd` to just use OpenOCD instead
    91  * installation instructions can be found [here](https://github.com/RIOT-OS/RIOT/wiki/OpenOCD)
    92 * Optional: picolibc for `arm-none-eabi` to link against picolibc instead of newlib
    93 * Optional: clang to build with `TOOLCHAIN=llvm`
    94 * Optional: GDB multiarch for debugging
    95  * If no multiarch package is available, use GDB for `arm-none-eabi` instead
    96 
    97 ### Architecture: Xtensa
    98 
    99 #### ESP32
    100 
    101 * @ref esp32_toolchain "Toolchain for ESP32"
    102 * [esptool](https://github.com/espressif/esptool) for flashing
    103 * Optional: OpenOCD and GDB (multiarch version) for @ref esp32_jtag_debugging "debugging via JTAG"
    104 
    105 #### ESP8266
    106 
    107 * @ref esp8266_toolchain "Toolchain for ESP8266"
    108 * [esptool](https://github.com/espressif/esptool) for flashing
    109 * Optional: GDB (multiarch version) for @ref esp8266_esp_gdbstub "debugging via the gdbstub"
    110  interface for the ESP8266
    111 
    112 ### Architecture: AVR
    113 
    114 * GCC and binutils for AVR and avrlibc
    115  * Alternatively: Install docker and export `BUILD_IN_DOCKER=1`
    116 * avrdude for flashing
    117 * Optional: AVaRICE and GDB (multiarch version) for debugging
    118 
    119 ### Architecture: RISC-V
    120 
    121 * GCC, binutils, and newlib for RISC-V (target triple should start with `riscv` and end with
    122  `-none-elf` or `-unknown-elf`. Note that most packages are multilib, e.g. `riscv64-unknown-elf`
    123  will likely work fine for 32 bit RISC-V boards)
    124  * Alternatively: Install docker and export `BUILD_IN_DOCKER=1`
    125 * OpenOCD for debugging/flashing (some new boards might require a patched version of OpenOCD or a
    126  recent build from the git sources)
    127 * Optional: picolibc to link against picolibc instead of newlib (recommended)
    128 * Optional: clang to build with `TOOLCHAIN=llvm`
    129 * Optional: GDB multiarch for debugging
    130 
    131 ### Architecture: MSP430
    132 
    133 * GCC, binutils, and newlib for MSP430
    134  * Alternatively: Install docker and export `BUILD_IN_DOCKER=1`
    135 * [mspdebug](https://github.com/dlbeer/mspdebug) for flashing/debugging
    136  * Optional: [MSP Debug Stack](https://www.ti.com/tool/download/MSPDS-OPEN-SOURCE) for additional
    137  board support
    138 * Optional: GDB multiarch for debugging
    139 
    140 ### Architecture: native
    141 
    142 * On 64 bit systems: multilib versions for your host compilers, standard C library, and development
    143  headers
    144  * Alternatively: Compile with `BUILD_IN_DOCKER=1`. Note that for running the executable you
    145  still need a multilib system (or 32 bit Linux) with glibc a standard C library.
    146 * A C library supporting the deprecated POSIX.1-2001 ucontext library (e.g. glibc, FreeBSD's libc)
    147 * Optional: GDB for debugging. (Prefer the multiarch version, this will also work for other boards)

    149 The build system {#the-build-system}
    150 ----------------
    151 RIOT uses [GNU make](https://www.gnu.org/software/make/) as build system. The
    152 simplest way to compile and link an application with RIOT, is to set up a
    153 Makefile providing at least the following variables:
    154 
    155  * `APPLICATION`: should contain the (unique) name of your application
    156  * `BOARD`: specifies the platform the application should be built for by
    157  default
    158  * `RIOTBASE`: specifies the path to your copy of the RIOT repository (note,
    159  that you may want to use `$(CURDIR)` here, to give a relative path)
    160 
    161 Additionally it has to include the `Makefile.include`, located in RIOT's root
    162 directory:
    163 
    164 ~~~~~~~~ {.mk}
    165 # a minimal application Makefile
    166 APPLICATION = mini-makefile
    167 BOARD ?= native
    168 RIOTBASE ?= $(CURDIR)/../RIOT
    169 
    170 include $(RIOTBASE)/Makefile.include
    171 ~~~~~~~~
    172 
    173 You can use Make's `?=` operator in order to allow overwriting
    174 variables from the command line. For example, you can easily specify the target
    175 platform, using the sample Makefile, by invoking make like this:
    176 
    177 ~~~~~~~~ {.sh}
    178 make BOARD=iotlab-m3
    179 ~~~~~~~~
    180 
    181 Besides typical targets like `clean`, `all`, or `doc`, RIOT provides the
    182 special targets `flash` and `term` to invoke the configured flashing and
    183 terminal tools for the specified platform. These targets use the variable
    184 `PORT` for the serial communication to the device. Neither this variable nor
    185 the targets `flash` and `term` are mandatory for the native port.
    186 
    187 For the native port, `PORT` has a special meaning: it is used to identify the
    188 tap interface if the `netdev_tap` module is used. The target `debug` can be
    189 used to invoke a debugger on some platforms. For the native port the additional
    190 targets such as `all-valgrind` and `valgrind` exist. Refer to
    191 `cpu/native/README.md` for additional information
    192 
    193 Some RIOT directories contain special Makefiles like `Makefile.base`,
    194 `Makefile.include` or `Makefile.dep`. The first one can be included into other
    195 Makefiles to define some standard targets. The files called `Makefile.include`
    196 are used in `boards` and `cpu` to append target specific information to
    197 variables like `INCLUDES`, setting the include paths. `Makefile.dep` serves to
    198 define dependencies.
    199 
    200 Unless specified otherwise, make will create an elf-file as well as an Intel
    201 hex file in the `bin` folder of your application directory.
    202 
    203 Learn more about the build system in the
    204 [Wiki](https://github.com/RIOT-OS/RIOT/wiki/The-Make-Build-System)

    206 Building and executing an example {#building-and-executing-an-example}
    207 ---------------------------------
    208 RIOT provides a number of examples in the `examples/` directory. Every example
    209 has a README that documents its usage and its purpose. You can build them by
    210 opening a shell, navigating to an example (e.g. `examples/basic/default`), and
    211 running:
    212 
    213 ~~~~~~~~ {.sh}
    214 make BOARD=samr21-xpro
    215 ~~~~~~~~
    216 
    217 or
    218 
    219 ~~~~~~~~ {.sh}
    220 make all BOARD=samr21-xpro
    221 ~~~~~~~~
    222 
    223 To flash the application to a board just run:
    224 
    225 ~~~~~~~~ {.sh}
    226 make flash BOARD=samr21-xpro
    227 ~~~~~~~~
    228 
    229 You can then access the board via the serial interface:
    230 
    231 ~~~~~~~~ {.sh}
    232 make term BOARD=samr21-xpro
    233 ~~~~~~~~
    234 
    235 If you are using multiple boards you can use the `PORT` macro to specify the
    236 serial interface:
    237 
    238 ~~~~~~~~ {.sh}
    239 make term BOARD=samr21-xpro PORT=/dev/ttyACM1
    240 ~~~~~~~~
    241 
    242 For flashing and accessing the board via the serial interface, the current user
    243 needs to have the correct access rights on the serial device.
    244 The easiest way to ensure this is to add the current user to the group that is
    245 owning the serial device. For example, this can be achieved on Linux by issuing
    246 the following line, logging out and logging in again:
    247 
    248 ~~~~~~~~ {.sh}
    249 sudo usermod -aG $(stat --format="%G" /dev/ttyACM0) $USER
    250 ~~~~~~~~
    251 
    252 Note that the `PORT` macro has a slightly different semantic in `native`. Here
    253 it is used to provide the name of the TAP interface you want to use for the
    254 virtualized networking capabilities of RIOT.
    255 
    256 We use `pyterm` as the default terminal application. It is shipped with RIOT in
    257 the `dist/tools/pyterm/` directory. If you choose to use another terminal
    258 program you can set `TERMPROG` (and if need be the `TERMFLAGS`) macros:
    259 
    260 ~~~~~~~~ {.sh}
    261 make -C examples/networking/gnrc/gnrc_networking/ term \
    262  BOARD=samr21-xpro \
    263  TERMPROG=gtkterm \
    264  TERMFLAGS="-s 115200 -p /dev/ttyACM0 -e"
    265 ~~~~~~~~
    266 
    267 You may not see the greeting
    268 
    269  main(): This is RIOT!
    270 
    271 when you flash the board. In this case, type `reboot` in the command line or reboot manually.

    273 Configuring an application {#configuring-an-application}
    274 --------------------------
    275 Many modules in RIOT offer configuration options that will be considered during
    276 compile-time.They are modeled as macros that can be overridden by the user.
    277 Currently there are two ways of doing this: using `CFLAGS` or via
    278 @ref kconfig-in-riot "Kconfig" (the last one is currently only possible for a
    279 subset of modules).
    280 
    281 For instructions on how to configure via `CFLAGS` check the
    282 @ref config "identified compile-time configurations". To learn how to use
    283 Kconfig in RIOT, please refer to the @ref kconfig-users-guide.
    284 
    285 Default configurations {#default-configurations}
    286 ----------------------
    287 When devices have a common access interface, having a default configuration to
    288 enable them across platforms, without having to explicitly specify which modules
    289 to include, comes in handy. For this, two pseudomodules are defined:
    290 
    291 - `saul_default`: will enable all the drivers of sensors and actuators that are
    292 present in the target platform.
    293 
    294 - `netdev_default`: will enable all the drivers of network devices
    295 present in the target platform.

    297 Use Docker to build RIOT {#docker}
    298 ========================
    299 [Docker](https://www.docker.com/) is a platform that allows packaging software into containers that can easily be run on any Linux that has Docker installed.
    300 
    301 You can download a RIOT Docker container from the Docker Hub and then use that to build your project making use of all toolchains that we've preinstalled in the container.
    302 
    303 It can be helpful to use Docker especially if you are working with ESP, since the
    304 installation might be easier this way.
    305 
    306 Setup {#docker-setup}
    307 -----
    308 
    309 ### Installing docker
    310 
    311 To use the RIOT docker build image, the Docker application needs to be installed on your system.
    312 To install Docker, depending on your operating system, use `sudo apt-get install docker` or a variant.
    313 
    314 The user on your computer requires permission to access and use docker. There are two ways to manage this:
    315 - Your OS distribution may create a group called `docker`. If so, then adding yourself to that group (and logging out and in again) should grant you permission.
    316 - Execute docker with sudo. This is in fact the most secure and recommended setup (see [here](https://docs.docker.com/install/linux/linux-postinstall/), [here](https://docs.docker.com/engine/security/security/#docker-daemon-attack-surface), [here](https://www.projectatomic.io/blog/2015/08/why-we-dont-let-non-root-users-run-docker-in-centos-fedora-or-rhel/) and [here](https://fosterelli.co/privilege-escalation-via-docker.html)). No extra setup steps are needed. `make` should be instructed to use `sudo` by setting `DOCKER="sudo docker"` in the command line.
    317 
    318 ### Downloading and testing RIOT docker container
    319 
    320 Finally, download the pre-built RIOT Docker container:
    321 
    322 ```console
    323 # docker pull riot/riotbuild
    324 ```
    325 
    326 This will take a while. If it finishes correctly, you can then use the toolchains contained in the Docker container:
    327 (**from the riot root**):
    328 
    329 ```console
    330 $ docker run --rm -i -t -u $UID -v $(pwd):/data/riotbuild riot/riotbuild ./dist/tools/compile_test/compile_test.py
    331 ```
    332 
    333 Usage
    334 -----
    335 
    336 The RIOT build system provides support for using the Docker container to build RIOT projects, so you do not need to type the long docker command line every time:
    337 
    338 (**from the directory you would normally run make, e.g. examples/basic/default**)
    339 
    340 ```console
    341 $ make BUILD_IN_DOCKER=1
    342 ```
    343 
    344 If your user does not have permissions to access the Docker daemon:
    345 
    346 ```console
    347 $ make BUILD_IN_DOCKER=1 DOCKER="sudo docker"
    348 ```
    349 
    350 to always use Docker for building, set `BUILD_IN_DOCKER=1` (and if necessary `DOCKER="sudo docker"`) in the environment:
    351 
    352 ```console
    353 $ export BUILD_IN_DOCKER=1
    354 ```
    355 
    356 running make without specifying `BUILD_IN_DOCKER=1` will still use Docker (because of the environment variable)
    357 
    358 Troubleshooting {#docker-troubleshooting}
    359 ---------------
    360 
    361 On some Ubuntu versions a make with `BUILD_IN_DOCKER=1` can't resolve the host name of for example github.com. To fix this add the file `/etc/docker/daemon.json` with the address of your DNS Server.
    362 
    363 For more details see @ref build-in-docker.

    365 Generating compile_commands.json e.g. for code completion in IDEs
    366 =================================================================
    367 
    368 A `compile_commands.json` for the selected board can be generated by running inside the application
    369 folder the following:
    370 
    371 ```console
    372 $ make compile-commands
    373 ```
    374 
    375 This target will honor the variables controlling the build process such as `BOARD`, `TOOLCHAIN`,
    376 `DEVELHELP`, etc. just like the usual build process. This works without actual compilation. By
    377 default, the `compile_commands.json` is placed in the RIOT base directory. This behavior can be
    378 overwritten using the `COMPILE_COMMANDS_PATH` variable by specifying the full absolute path
    379 (including file name) of the `compile_commands.json` instead.
    380 
    381 ***Note:*** By default, the built-in include search directories of GCC will be explicitly added
    382 and flags incompatible with `clangd` will be dropped. This will allow using `clangd` as language
    383 server out of the box. If this is not desired, run `export COMPILE_COMMANDS_FLAGS=""` to turn
    384 modification of the compile commands off. For a list of available flags, run
    385 `./dist/tools/compile_commands/compile_commands.py --help` in the RIOT base directory.

    387 Using the native port with networking
    388 =====================================
    389 
    390 If you compile RIOT for the native cpu and include the `netdev_tap` module,
    391 you can specify a network interface like this: `PORT=tap0 make term`

    393 Setting up a tap network
    394 ------------------------
    395 
    396 There is a shell script in `RIOT/dist/tools/tapsetup` called `tapsetup` which
    397 you can use to create a network of tap interfaces.
    398 
    399 *USAGE*
    400 
    401 To create a bridge and two (or `count` at your option) tap interfaces:
    402 ~~~~~~~{.sh}
    403  sudo ./dist/tools/tapsetup/tapsetup [-c [<count>]]
    404 ~~~~~~~
    405 
    406 A detailed example can be found in `examples/networking/gnrc/gnrc_networking`.
